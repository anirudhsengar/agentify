1. Project Overview

agentify is a standalone TypeScript ESM CLI published to npm. The single public command (agentify) installs & runs without subcommands, with no Pi-extension manifest, and turns any repository into an "agentic codebase" through:

  1. Audit (brownfield): in-process Pi session explores the repo, fills a 10-dimension codebase map, and only when every coverage dimension is covered, TypeScript renderers transactionally apply deterministic agentify:managed-marked files: AGENTS.md, specs/README.md, ai_docs/README.md, feature specialists under .pi/agents/, project workflow specs under .pi/workflows/, expert directories under .pi/prompts/experts/, repo-specific skill candidates under .pi/skills/, and conditional docs.
  2. Formation (greenfield): a local-first chat collects typed formation data via the write_greenfield_artifacts tool, agentify deterministically renders CONTEXT.md, GOALS.md, PRDs/plans/issues/specs from it. A stop_at hard gate rejects artifacts beyond the user-approved milestone, and substance gating blocks scaffold install if planning artifacts are placeholder-thin.
  3. Harness export + GitHub stamp: copies skills into .agents/skills/ and .claude/skills/; renders Codex (.codex/agents/*.toml) and Claude (.claude/agents/*.md, CLAUDE.md) exports; stamps the GitHub Actions runtime under .github/ so the post-bootstrap async loop is GitHub-issues-driven.

After bootstrap, issues labeled agent:queued and agent:implement trigger trusted GitHub Actions workflows that run a credential-free Pi orchestration planner, capture issue context, branch, implement, and open a draft PR labeled agent:review. Review workflow approves or requeues, drill workflow triages post-launch GitHub issues, refresh workflow re-emits the agentic surface on every default-branch push.

Two mode of entry, one shipped loop. The public orchestration plane is GitHub Actions (ADR 0015). Webhook, AIW, Orchestrator and Coms modules ship as internal library code, not as separate public commands.

2. CLI Command Surface

The runtime entry point is agentify with no positional arguments. Config-utility subcommands (login, logout, models) operate only on ~/.agentify/{config,auth}.json — they never invoke the audit runtime (ADR 0008 amendment, 2026-07-09). The full surface is:

# Runtime
agentify [options]
    -h, --help                 Show help and exit.
    -v, --version              Print version and exit.
    --mode <kind>              Skip classification for ambiguous repos.
                               <kind> ∈ brownfield|greenfield.
    --targets <csv>            Skip the agent-target picker. Comma-separated
                               agent IDs (e.g. 'claude-code,codex,cursor').
                               Skips the picker for non-interactive use;
                               persisted targets are NOT respected (ADR 0018).

# Config-utility subcommands (no audit runtime invoked)
agentify login [--provider <name>] [--key <key>]
agentify logout [--provider <name> | --all] [--yes]
agentify models list [--provider <name>]
agentify models show [--resolved]
agentify models set <provider/model>
agentify models set <slot> <provider/model>
agentify models unset [<slot>]

Implementation lives in src/cli.ts. Subcommand dispatch runs before --mode parsing so that flags belonging to a subcommand (e.g. models list --provider x) are not consumed by the top-level parser. The first positional argument selects the dispatch lane:

// from src/cli.ts
if (argv.length > 0) {
  const head = argv[0];
  if (head === "login" || head === "logout" || head === "models") {
    await dispatchSubcommand(argv, subcommandCtx);
    return;
  }
  throw new Error(
    `unknown subcommand '${head}'. Known subcommands: login, logout, models. ` +
      `Run \`agentify --help\` for usage.`
  );
}

If the runtime lane is selected, runAgentifyApp() in agentify-app.ts inspects repo state via inspectAgentifyRepoState() and either:

  • Returns an attach report (mode already initialized, just summarises feature_agents, workflows, experts, skills + last log path + GitHub readiness).
  • Returns a partial recovery report (lists missing files).
  • Forwards to runAgentify() in run-agentify.ts for a real run.

2.2 --targets <csv> (ADR 0018)

Comma-separated registry IDs (e.g. claude-code,codex,cursor). Validated against the agent registry — unknown IDs throw with a clear message. Skips the picker for non-interactive use; persisted targets are not respected (the picker always re-prompts). The split into premium (codex | claude | pi) and additional agents is automatic via getPremiumTargets() in src/core/agent-registry.ts.

2.3 Interactive UX

ConsoleUi implements the AgentifyUi interface (src/core/types.ts): status(), info(), error(), promptSelect(), promptMultiSelect(), promptSecret(). The secret prompt disables echo by switching stdin into raw mode, supports Backspace, and rejects Ctrl-C with "Interrupted."

promptMultiSelect accepts comma-separated numbers (1,3,5), all, or none. It is used by promptTargets() in target-picker.ts to surface the full agent registry (Claude Code / Codex / Pi pre-selected; Cursor, OpenCode, Windsurf, Cline, Continue, Roo Code, Kilo Code, GitHub Copilot, and 60+ more behind them).

When stdin is not a TTY (e.g. CI), the UI throws a precise error if anything tries to prompt:

Cannot prompt because stdin is not interactive. Pre-configure auth via a provider env var
  (e.g. OPENAI_API_KEY / ANTHROPIC_API_KEY) or ~/.agentify/auth.json before running agentify.

6. CLI Implementation (src/cli.ts)

src/cli.ts is a TypeScript module loaded at runtime by bin/agentify.js through jiti. Highlights:

  • Imports: node:fs, node:path, node:readline/promises, node:process, defaultConfigDir, PiSdkRuntime, packageRoot, runAgentifyApp, dispatchSubcommand + printSubcommandHelp from cli-commands.ts, isKnownAgent from agent-registry.ts, type AgentifyUi.
  • Class ConsoleUi implements AgentifyUi with raw-mode secret prompt that catches Ctrl-C (0x03), Enter/Return, Backspace (0x7F). Also implements promptMultiSelect for the target picker (comma-separated numbers / all / none).
  • Reading version: readPackageVersion() parses package.json from packageRoot() in pi-sdk-runtime.ts.
  • Top-level dispatch order:
    1. --help / -h → printHelp() (which also calls printSubcommandHelp()).
    2. --version / -v → print version.
    3. Subcommand dispatch (BEFORE --mode parsing): if argv[0] ∈ {login, logout, models} → dispatchSubcommand(argv, ctx) and return. Anything else throws: "unknown subcommand '<x>'. Known subcommands: login, logout, models."
    4. --mode <kind> parsed, value validated as brownfield | greenfield.
    5. --targets <csv> parsed, each ID validated via isKnownAgent (unknown IDs throw with the bad name).
    6. Delegate to runAgentifyApp({ args, cwd, ui, runtime, mode, targetsOverride }).
  • Defensive guard (in agentify-app.ts): if args.length > 0 after the subcommand lane is selected, throws "agentify does not accept '<x>'. Known subcommands: login, logout, models."
  • Self-invocation: if (import.meta.url === 'file://' + process.argv[1]) main().catch(...) — jiti-imported calls must use the exported main().

7. Bootstrap Orchestration

src/core/agentify-app.ts is the public seam; src/core/run-agentify.ts does the actual work. Flow:

7.1 runAgentifyApp(options)

  1. Rejects positional args.
  2. inspectAgentifyRepoState(cwd, configDir) — inspects manifest + filesystem; returns {mode, status, missing, found, counts}.
  3. If status === "ready" → attachToInitializedRepo(): prints status, surface counts, last log path, GitHub readiness — exit (does not re-audit).
  4. If status === "partial" → prints recovery banner with missing and previous run info, then forwards to runAgentify().
  5. Otherwise → forwards to runAgentify().

7.2 runAgentify(options)

10. Defense-in-Depth Hook (src/core/audit/defense-hook.ts)

The hook is constructed via makeDefenseHook({ agentDomain, repoJail, protectedPaths }). It fires for the builder, greenfield, and every explorer sub-agent. Hook only runs when an agentify session is marked active, or repoJail is set, or an agentDomain is set.

Layer A — Bash compound-operator rejection

SHELL_OPERATORS_REGEX = /(&&|\|\||;|\||`|\$\(|>|<)/

Anything matching this (in any position) is blocked with reason "defense: compound bash command rejected". Stops npm test && rm -rf even when the prefix matches a whitelist.

Critical rule: The blacklist must NOT whitelist bare python / node / bash. They must be pinned to specific scripts only. Otherwise a context-decayed LLM escapes into arbitrary code execution.

Layer A.3 — Script-content scanner

When the bash command is <runner> <file> (e.g., python cleanup.py), the hook reads the first 64 KB of the script and re-runs the bash blacklist against its contents. Closes the "Marquee Break" attack pattern (write cleanup.py && python cleanup.py).

Layer B — Zero-access path guard (read/write/edit)

ZERO_ACCESS_PATH_REGEX (with negative lookahead for .env.sample, .env.example, .env.template):

/(\.env(?!\.sample(?:.*)?$|\.example(?:.*)?$|\.template(?:.*)?$)
   |secrets\.(?!sample(?:.*)?$|example(?:.*)?$|template(?:.*)?$)
   |~?\/?\.ssh\/
   |\/etc\/
   |\.aws\/credentials
   |\.npmrc$
   |id_(rsa|ed25519|ecdsa|dsa)(\.pub)?$)/

The hook uses extractPathFromInputForTool(tool, input) with PATH_PARAM_NAMES lookup so the LLM can't slip a path past via an alternate key (path, filePath, filepath, filename, file, file_path, outputPath, output_path, destination, target).

Layer B.2 — Agentify credential-store protection

AGENTIFY_HOME = path.resolve(os.homedir(), ".agentify") — never readable or writable by the agent, even via repo symlinks. The realResolve() helper follows symlinks along the nearest existing ancestor so a repo file that symlinks to ~/.agentify/auth.json is still blocked. A repo-relative .agentify/ is OK (the example .agentify/webhooks.example.json ships as a sample config).

Layer B.3 — User-owned file protection

Pre-existing unmanaged files from the ownership snapshot are passed as protectedPaths; any write that targets one of them is blocked: "defense: '<path>' is a user-owned file; agentify will not overwrite it"

Layer B.4 — Repository jail

When repoJail: true (the builder, greenfield, and explorer sub-agents), every write/edit target must resolve inside realpath(cwd). "defense repo-jail: write on '<x>' resolves outside the repository"

Layer C — create_agent depth cap

When create_agent is invoked and activeTools does not include it, the call is blocked — sub-agent sessions never have it in their allowlist, so a depth≥2 attempt is rejected. "defense depth-cap: create_agent is reserved for the orchestrator session"

Layer D — escalate_to_orchestrator depth cap

Same shape: only allowed when the session's activeTools includes escalate_to_orchestrator (parent is the live orchestrator). "defense depth-cap: escalate_to_orchestrator is reserved for orchestrator-spawned sub-agents"

Layer E — Domain lock

For orchestrator sub-agents that declare a domain (path globs):

  • agentDomain === null — no constraint (backward compat)
  • agentDomain === [] — read-only
  • agentDomain = ["docs/**", ...] — only writes matching a glob are allowed

Glob match (no minimatch dep) supports **, *, ?, and absolute patterns.

22.1 Why packaged/skills/ (ADR 0002 + post-pivot)

Before the pivot, the maintainer repo carried .agents/skills/ + a .claude/skills/ symlink mirror at the root. Any coding harness the maintainer opened the repo in would auto-load every shipped skill. The shipped-paths.ts constant SHIPPED_SKILLS_SUBDIR = path.join("packaged", "skills") is the single source of truth.

39. Named Model Slots — primary | explorer | scoring (ADR 0017)

agentify has at least seven distinct LLM invocation sites (audit builder, greenfield chat, AIW phases, orchestrator host, sub-agents, webhook workers, expert driver). Before ADR 0017, every site resolved to a single (provider, model) pair from config.json, and spawn_explorer's MODE_MODEL_DEFAULT table (topography → haiku, gap_filler → opus, …) was advisory only — written into the sub-agent's "Constraints" prompt paragraph but never actually routed the call to a different Model<Api>. The sub-agent ran on the parent's resolved model regardless. That was a documentation lie; the table promised behavior the code didn't deliver.

ADR 0017 introduces three named slots with a 4-tier resolver and a "max quality is the floor" invariant: unset slots fall back to primary; explicit user choices are never silently overridden.
